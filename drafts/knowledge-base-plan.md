# Research Knowledge Base Development Plan

*toolkit.socialnetwork.health — research corpus for humans and AIs*
*Drafted June 2026*

## Purpose

Turn a ~500-paper research corpus (plus additional references) into a knowledge base that serves two consumers from one substrate:

1. **Humans** — fast re-orientation at the wiki. Mouse over a reference, recover in ~5 seconds what the paper claims and why it's in the corpus. The cards amortize understanding; they never replace it. Understanding is built when reading papers and when correcting wrong cards — the card is the externalized trace that lets a future reader skip the re-derivation.
2. **AIs** — evidence synthesis. Given a claim, identify the right 5–15 papers fast, then read them in full. Whole-document agentic reading over clean files, not chunk-based RAG.

## Architecture in one paragraph

A GitHub repository of flat markdown files is the canonical store: one file per paper, YAML frontmatter card on top, clean extracted full text below. Everything else is **derived and regenerated by CI on every push** — the catalog, the trimmed card compendium, and a JSON export of all cards. Consumers are thin adapters over these artifacts: wiki hover-cards render the JSON; a Claude Project holds the compendium; a remote MCP server (plus downloadable bundle) exposes search/read tools; a Claude Code skill teaches agents the repo conventions. The old spreadsheet seeds the reference list and then retires. The old 50-word AI summaries are not migrated.

## Card schema (starting draft — refine in pilot)

```yaml
id: doi-or-fallback-id
title: ...
authors: [...]
year: 2021
doi: 10.xxxx/xxxxx
citation: "Author et al. (2021). Title. Journal."
gist: >
  One sentence, in our framing (not the authors'), of what this
  paper contributes to our work.
claims:
  - text: "Atomic, plain-language claim."
    evidence: longitudinal   # rct | longitudinal | cross-sectional | qualitative | theoretical | review
  - text: "Second key claim."
    evidence: cross-sectional
population: "Who/where/when the evidence comes from."
limitation: "The one honest caveat a reader should know."
tags: [help-seeking, relationship-strength]   # controlled vocabulary
source: papers/doi-or-fallback-id.md
```

Design rule: the `gist` + `claims` block **is** the hover-card. The full card is what AIs consume. One schema, two readers.

## Phases

### Phase 1 — Schema design (~1 day)
Draft the card schema (above is the starting point). Decide what the 5-second hover read must contain. Define the initial goal-oriented tag candidates loosely; the controlled vocabulary is finalized in Phase 6.
**Done when:** schema is written down and the team agrees the gist+claims block answers "why is this paper here?"

### Phase 2 — Pilot on 10 papers (~1 day)
Pick ten papers the team knows well, ideally already cited on the wiki. Generate cards with Claude against the schema. Run both acceptance tests:
- **Human test:** can you re-orient from the gist+claims block in ~5 seconds?
- **AI test:** given only the ten cards, can Claude correctly identify which papers bear on a claim we care about?

Iterate the schema until both pass. Every flaw caught here is one not fixed 500 times later.
**Done when:** both tests pass and the schema is frozen for v1.

### Phase 3 — Corpus consolidation & canonical IDs (~a few days, mostly automated)
Run the full reference list through the paper-resolver stack (OpenAlex, Crossref, Unpaywall, etc.) for canonical metadata and DOIs. Dedupe. Assign one ID per paper (DOI; deterministic fallback where none exists). Stand up the GitHub repo as canonical store. Spreadsheet becomes a seed input, then retires.
**Done when:** every reference has exactly one ID, one (possibly empty) file, and canonical metadata.

### Phase 4 — Wiki-cited subset, end-to-end first (~1–2 weeks elapsed, light effort)
For the subset actually cited (or about to be cited) on the wiki — likely 50–150 papers: extract PDFs to clean markdown, generate cards, **human spot-check most of them**. PDF-to-markdown quality is the silent killer of everything downstream; check extraction here too.
**Done when:** every wiki-cited reference has a verified card. Value ships here, not at the end.

### Phase 5 — Batch the long tail (~a weekend of compute, light review)
Extract and card the remaining papers via a Claude Code batch job. Spot-check ~10%, weighted toward papers whose claims look surprising or load-bearing. A card that smells wrong on mouseover is a prompt to open the paper — which serves the understanding goal.
**Done when:** all ~500 papers have cards; spot-check error rate is acceptably low.

### Phase 6 — Derived artifacts & tag normalization (~1–2 days)
- Script generation of `CATALOG.md` (one or two lines per paper: title, year, tags, why-it's-here), the trimmed compendium (~100–150 tokens/card for hosted-assistant context limits), and `cards.json` for the wiki.
- Wire these into GitHub Actions: regenerate on every push. **Derived artifacts are never hand-edited.**
- Review the tag vocabulary that emerged across all cards; normalize to a controlled set of ~15–30 tags mapped to actual research questions. Consolidating after generation beats specifying perfectly upfront.

**Done when:** CI is green, artifacts regenerate on push, and tags are a closed vocabulary.

*Note: CATALOG.md effectively exists from Phase 2 onward (it's a script over whatever cards exist). Phase 6 just hardens the automation.*

### Phase 7 — Wire the consumers (order = priority)
1. **Wiki hover-cards** — wiki pulls `cards.json` nightly (or on webhook) and renders the gist+claims block on reference mouseover. The wiki never stores card content; it only renders it.
2. **Claude Project** — load the trimmed compendium as project knowledge for everyday synthesis questions; it answers from cards and names which papers to pull for depth.
3. **Remote MCP server** (primary) at e.g. `mcp.toolkit.socialnetwork.health` — thin FastMCP wrapper, ~4 tools: `search_catalog`, `get_card`, `get_fulltext`, `grep_corpus`. One URL anyone can paste into Claude, ChatGPT, or an agent framework; always current with last night's build.
4. **Downloadable MCP bundle** (secondary) — packaged local server (MCPB-style) from the website, for the local-first / offline audience. Same code, different wrapper.
5. **Claude Code skill** — a short skill file: "read CATALOG.md first; claims live in frontmatter; cite via the DOI field."
6. **Agent-readability cheap wins** — `llms.txt`, clean semantic HTML, raw-markdown endpoint per wiki page. The public GitHub repo itself is a distribution channel: agents and humans can simply clone it.

### Phase 8 — Intake ritual (~1 day to build, permanent payoff)
One command: given a DOI or PDF → resolve metadata → extract → generate card → commit → CI regenerates everything → wiki picks it up that night. If adding a paper takes 30 seconds, the corpus stays alive; if it takes manual spreadsheet rows, intake stops within a month.

## Deliberately deferred (revisit after a few weeks of real use)

- **Cross-paper claims index** (claim → supporting/contradicting papers). The natural v2, derivable from the cards at any time. Distinct from the catalog, which is the per-paper map and is required from day one.
- **Vector database / hybrid retrieval.** At ~500 docs, catalog + grep + full-document reads is simpler, more debuggable, and usually more accurate. If a query API later needs ranking, add hybrid BM25 + vector then.
- **WebMCP.** Real and well-backed (Chrome early preview since Feb 2026, W3C, Google/Microsoft co-authored) but experimental; broad support is realistically 2027, and currently only Gemini in Chrome consumes it. Its sweet spot is transactional sites, not read-and-understand wikis — the remote MCP server gives agents strictly more, in every client, today. **Trigger to build:** the declarative API stabilizes and more than one mainstream agent consumes WebMCP tools. Cost when triggered: days — a small page script registering `search_catalog` / `get_card` against the same JSON the hover-cards use.

## Operating principles

- One canonical substrate (the repo); every consumer is a thin, derived renderer — including access layers that don't exist yet.
- Derived artifacts are generated, never hand-edited; CI keeps them from drifting.
- Schema before scale; wiki-cited papers before the long tail; real usage tests before batch jobs.
- Cards amortize understanding; spot-checking is where the understanding gets built, not overhead.
- Ship v1, let actual queries decide what earns v2.
